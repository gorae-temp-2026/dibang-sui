# 이슈 작성 규칙

GitHub 이슈를 만들 때 따르는 가벼운 컨벤션. 라벨 체계·템플릿 파일은 두지 않고, 제목 접두 + 본문 골격으로만 운영한다.

## 1. 제목

일반 작업은 그대로 (예: `라운지 V2 카드 레이아웃 정리`).
특수 상황은 **접두 마커**로 구분한다.

| 접두     | 용도                                  |
|----------|---------------------------------------|
| `[임시]` | 되돌릴 변경 (기능 미준비로 인한 숨김 등) |
| `[버그]` | 동작 이상                              |
| `[제안]` | 기능 아이디어                          |
| `[리서치]` | 조사·실험                            |

같은 마커가 3건 이상 쌓이면 정식 라벨로 승격 검토.

## 2. 라벨

라벨은 **필요할 때 만든다.** 선제 생성하지 않는다.
승격 시 후보:
- 유형: `bug`, `enhancement`, `temp`, `research`
- 앱 스코프: `app:dibang-wedding`, `app:guest-web`, `app:admin`, `app:api`
- 우선순위(필요 시): `P0`, `P1`, `P2`

## 3. 본문 골격

### 공통 (모든 이슈)
- **배경** — 왜 이 이슈가 필요한가
- **변경 범위** — 대상 앱·경로·방식
- **체크리스트** — 작업 단위 todo

### `[임시]` 추가
- **복구 조건** — 언제 되돌리는가
- 코드에 `// TEMP: <설명> — issue #N` 주석 필수
- `rg "TEMP:"`로 미해결 임시 변경 전수 조사 가능

### `[버그]` 추가
- 재현 절차 / 기대 동작 / 실제 동작 / 환경

### `[제안]` 추가
- 문제 / 제안 / 대안 / 영향 범위

## 4. Close 조건

- 작업 PR이 머지되면 자동 close (PR 본문에 `Closes #N`)
- `[임시]`는 **복구 PR 머지 시점에 close.** 임시 변경을 적용한 PR 머지 시점에 닫지 않는다.

## 5. 첨부

스크린샷·이미지는 `gh` CLI로 직접 첨부가 안 되므로, 이슈 생성 후 GitHub 웹 UI에서 드래그앤드롭으로 붙인다.
