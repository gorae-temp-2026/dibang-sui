## 0. 공통 모델

- **event_type** ∈ `{ wedding, inyeon, (party, seungbu …) }`
- **event** = event_type의 한 인스턴스.
  - wedding event = **한 번의 결혼식**
  - inyeon event = **한 번의 매칭(matching)**
- **action** = raw 동사 (기계적 행위, 저장). **signal** = `project(action × event_type × 역할쌍)` 출력(저장 안 함).
- **signal 대분류 = 관계운영논리 CS · EM · AR · MP** (+ 소분류: 부조·대여·증여·유대…). 표기 `EM·CS·AR·MP` 각 ○★/△/✗.
- **신용 / Moi Credit = 다운스트림(3·4층) 결과**, signal 칼럼 아님. (EM·CS는 엣지로 전파, MP는 이행률로 3층 직접)

## 1. 두 카탈로그 요약

| | 웨딩 (event=결혼식) | 인연 (event=매칭) |
|---|---|---|
| 역할 | 혼주·하객·주례·업체 등 (host/guest 구조) | initiator·receiver·viewer·peer (방향 역할쌍) |
| 본진 signal | **EM**(부조) + CS 유대 | **CS**(이음) + 약간 EM(선물) |

| AR | 주례 선언(공인) | 없음 |
| event_id | 결혼식 id | matching_id |

## 2. 2층 진입 — 웨딩·인연이 한 그래프로

- 두 event_type의 액션이 **같은 노드(모이)** 위에 signal을 쌓는다: 웨딩=EM 중심, 인연=CS 중심.
- **②→③ 승급(인연↔웨딩 연결):** 인연 매칭(온라인 이음) 후 두 사람이 같은 wedding/party event에 재등장하면 인연 CS 엣지가 **오프라인 검증**됨 → 소속·중심 네트워크 공개 + 같은 네트워크 연결. (인연 event ↔ 웨딩 event 공유 노드로 fold가 이어짐)
- 결과: **2층 fold 입력의 event_type이 wedding·inyeon 둘 다** → 이후 3층 Φ로 노드별 Moi Credit 산출.

## 2-1. 강도 가중 — 매체(E4) 〔신규 260619〕

- project는 signal 유무뿐 아니라 **강도**도 정하며, 거기에 **매체(E4) 가중치 `w(E4)`**: **대면 > 동기온라인 > 비동기온라인** (Collins: co-presence가 유대에 결정적).
- **웨딩(대면) 유대 > 인연(동기온라인) 이음** — 같은 "관계 형성"도 매체로 강도가 다름. 주로 **CS에 작용**(EM 금액은 매체 무관).
- **②→③ 승급 = `w` 상승 사건**: 인연 온라인 이음이 오프라인 재회로 '대면' 승급 → CS 강도↑.
- `w(E4)` 잠정값(튜닝 대상).
