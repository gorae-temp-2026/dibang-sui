# 07. 2층 동역학 — fold (신호 → 상태)

> 1층(분류)이 "무엇이 있나"의 정적 사진이라면, 2층은 **신호들을 시간에 따라 누적해 관계·집단 상태를 만드는** 단계다(H8). fold가 그 누적 연산. 입력은 이벤트가 아니라 **신호**(project 출력, H11)다.
>
> 이 문서 = 상태 스키마(F2-1) + fold 연산(F2-2) + 결혼식 워크드 예제(F2-4) + 일관성(F2-6). 검증 스크립트로 전 항목 통과(2026-06-16).

## 1. 상태(state) 스키마 — fold의 출력

신뢰망의 누적 상태. 노드 = entity(사람|집단). 엣지 = 방향 있는 누적. **두 원장으로 갈린다**(H12 보강: 차이는 청산 유무):

| 원장 | 저장 | 조회 | 대수 |
|---|---|---|---|
| **EM 원장**(호혜) | 방향별 `gave[A→B][자원]` 누적 | `net(A,B,자원) = gave[A→B] − gave[B→A]` | (ℤ,+) **아벨군** — 역원=되갚음=청산 |
| **CS 원장**(유대) | 방향별 `tie[A→B][자원]` 누적 | `tie(A→B)` 그대로 (양방향 독립) | 방향별 (ℤ,+) **군**(셀마다 역원 있음) — 청산 없음 = **방향 간 상쇄 없음** |
| (거래) | 저장 안 함 | — | 즉시청산 → 영속 상태에 0 기여 |

- **EM net > 0** = 뒤(B)가 앞(A)에게 미청산 채무. 그 미청산 잔액이 곧 신뢰의 정량 흔적(H3).
- **CS는 방향별 독립**: `tie(A→B) ≠ tie(B→A)` 가능 — 같은 함께함 사건도 각자 증분이 다름(B층위, H12 보강). 상쇄가 없어 되갚아도 0이 안 됨.
- **관계(A,B)** = 둘 사이 `EM net` + `CS tie 양방향`의 묶음. → 이 상태를 **R축으로 분류**할 수 있다(R2 운영장부 = 어느 원장이 지배적인가). 이것이 H10의 "R축 튜플 = fold 출력 상태의 정지컷".
- **집단** = N개 노드의 원장 묶음(H8: 집단 = N인 관계).

### 1-1. PHI-4 결정 (2026-06-17, 완료) — 각 자원 Hom값의 대수와 이행의 처리

3층 enriched category의 베이스 V(엣지값이 사는 대수)를 자원별로 확정. 단일 베이스 통합은 안 함(자원별 인정, 통합은 점수 단계=PHI-2로 미룸).

- **EM 호혜잔액 = (ℤ,+) 아벨군.** 한 쌍에 반대칭 단일 net값(`net(A,B) = −net(B,A)`). 역원 = 되갚음 = 청산.
- **CS 유대 = 방향별 (ℤ,+) 군.** 각 방향 셀이 군이라 역원이 있고, 한 칸 *안에서* 음수 신호(배신)로 그 방향 유대가 0·음수까지 식을 수 있음.
- **정정 (이전 "CS=역원 없는 모노이드" 폐기):** EM/CS 차이는 "군 vs 모노이드"가 아니다 — **둘 다 군**이다. 차이는 *구조*다. EM은 두 방향을 net 한 칸으로 빼서 합치므로 **방향 간 상쇄(=청산)**가 일어나고, CS는 두 방향을 독립 두 칸으로 두므로 **방향 간 상쇄가 없다(청산 없음)**. 즉 "독립 두 칸"과 "청산 없음"은 같은 사실의 두 표현. CS엔 갚을 빚-장부가 없어서(H3) 청산이라는 사건 자체가 성립하지 않는다.
- **선형 단서:** 한 칸 안의 "배신이 호의로 캔슬 안 됨"은 선형(덧셈)에선 −5+5=0으로 *사실 캔슬된다*. 안 캔슬되게 하려면 비선형(배신 가중, §5)이 필요 — 1차는 선형 유지.
**이행(대여상환·MP 대금·약속)의 처리 = node 속성 + EM 엣지 전파 (자체 베이스 없음).** 이행은 EM·CS와 성격이 달라 따로 끝까지 풀어 적는다.

**(가) edge냐 node냐 — 베이스를 묻기 전에 먼저 풀어야 할 게이트.** EM·CS는 본질이 엣지다(두 사람 사이의 방향 잔액·유대라, 노드로 내리면 "누구와 엮였나"라는 그래프 구조가 사라져 신용 모델이 무의미해진다). 그러나 이행은 둘 다 가능하다 — 한 사람의 "약속 지키는 정도"를 하나의 점수로 접으면 node, 연결별로 "철수→영희 갚음 / 철수→민수 떼임"을 보존하면 edge. 그래서 이행에 어떤 대수(베이스)를 줄지 묻기 *전에*, "이행이 애초에 그래프 안(엣지)이냐 밖(노드)이냐"라는 게이트부터 정해야 한다.

**(나) 결정 = node.** 이행 결과는 사람마다 하나의 신뢰도 스칼라 `r(사람)`으로 둔다(약속·대금·상환을 지킨 정도). 이는 "신용 = 본인 행적"이라는 개인 신용점수 직관과 맞고, MP·대여 default가 이미 "2층 fold를 건너뛰고 3층으로 직접 간다"(§2-2, 05 H12 보강2)고 정한 처리와도 정합한다 — 이행은 2층 잔액이 아니라 3층 노드 특성으로 도착한다.

**(다) 그런데도 리스크는 전파된다 — 단, 새 '이행 엣지'가 아니라 이미 있는 EM 노출 엣지를 탄다.** 누가 누구에게 돈·보증을 걸어놨나(EM 잔액)가 곧 리스크가 흐를 통로다. B가 default하면 `r(B)`가 낮아지고, 그 낮아진 신뢰도가 EM 그래프를 따라 B에게 노출된 이웃(채권자)에게 확산된다. 이는 "노드 속성 + 그래프 확산"이라는 표준 구성(개인화 PageRank가 노드 특성을 엣지 따라 섞는 것과 같은 모양)이며, 05 H12(4)의 "신용 = 본인 행적(이행 node) + 네트워크 전파(EM 그래프)"를 그대로 구체화한 것이다.

**(라) 그래서 이행은 자체 베이스가 필요 없다.** 이행이 node라서 enrichment(엣지값) 베이스를 따로 가질 이유가 없다. 전파는 EM 베이스(아벨군) 그래프 위에서 일어나므로, 이행은 그 위를 흐르는 노드 신뢰도일 뿐이다. 덤으로, default를 음수 엣지로 만들어 PageRank류에 투입할 때 생기는 수렴 불안정(음수 가중치)도 피한다 — 경계 있는 노드 신뢰도(예: 0~1)를 양수 노출 엣지 위로 확산시키는 편이 안정적이기 때문이다.

**(마) "노출 0인데 평판으로만 엮인 사람"은 안 닿는 게 맞다 — 버그가 아니라 기능이다.** EM 엣지가 없으면 리스크가 안 번지는데, 이는 신용 점수에서 *올바른* 동작이다. 안 닿는 집합(순수 엮임·금전 노출 0)과 닿으면 부당한 집합(연좌제)이 같은 집합이기 때문이다 — 안 번지는 것이 곧 공정성이다. 게다가 리스크 전염을 정당화할 만큼 깊은 엮임(동업 등 공동 출자·연대보증·공동 채무·상호 대여)은 거의 항상 EM 흔적을 남긴다("돈거래 0인 동업자"는 사실상 형용모순). 따라서 현실에서 EM 노출 그래프는 리스크 목적엔 사실상 완전하다. 유일한 진짜 예외인 사기링(고의로 돈자국 없이 엮인 패거리)은 신용 점수가 아니라 별도 사기탐지 레이어(기기·행동 동시발생 등)의 몫이며, 공정성·해석을 지키기 위해 신용 모델과 분리해 둔다.

**(바) 다른 PHI에 남기는 것.** 이행 리스크 전파의 *방향*(default한 사람 → 채권자 쪽 = EM 채권 방향)은 PHI-1에서, 본인 신뢰도와 전파받은 리스크의 *합산 비중*은 PHI-2·PHI-3에서 정한다. 이행 자체는 PHI-4에서 "node·별도 베이스 없음"으로 종료한다.

> **PHI-4 종료:** enrichment 베이스 목록 = { EM 호혜잔액 = (ℤ,+) 아벨군, CS 유대 = 방향별 (ℤ,+) 군 } 둘뿐. 이행은 엣지값이 아니라 node 신뢰도(EM 그래프 위로 전파)라 자체 베이스를 갖지 않는다. → 다음 선결은 PHI-1(전파 방향).

## 2. fold 연산

신호 `{from, to, 자원, 크기, 청산, default판정}` 리스트 → 상태. (청산: 미청산=EM / 무청산=CS / 즉시=거래 · default판정: 미반환이 신용 손상 사건인가 — 부조 불가/대여 가능, §2-2)

```
fold(signals):
  state = { EM:{}, CS:{} }
  for s in signals:
    if s.청산 == 즉시:        continue               # 거래 → 트러스트 0(단 대여 default는 3층 직접)
    bucket = (s.청산 == 미청산) ? state.EM : state.CS
    bucket[(s.from, s.to, s.자원, s.default판정)] += s.크기   # 키에 default판정 포함 → 부조·대여 분리(§2-2)
  return state
# EM 조회: net(A,B,r,d) = EM[(A,B,r,d)] − EM[(B,A,r,d)]   (같은 종류끼리만 청산)
```

- **순수 부호 덧셈** → 결합·교환법칙 성립 → **순서 무관**(모노이드/아벨군). 검증 ✅.
- **EM만 청산**: 반대 방향(되갚음)이 와서 net을 0으로 당김(아벨군 역원). CS는 청산 규칙이 안 붙음 — **하나의 fold 틀 + EM에만 청산**(H12 보강).
- 자원종류가 파라미터. 같은 fold가 돈·유대·노동을 각자 원장에 누적.

### 2-2. fold 키의 원자 차원 — "같은 자원, 다른 종류는 따로 쌓는다" (2026-06-16)

같은 돈 10만이라도 **축의금(부조)·빌려줌(대여)·선물(증여)**은 신뢰적으로 다르다 → 한 칸에 합산하면 안 된다. 그런데 키에 **"호혜유형(부조/대여/증여)"이라는 복합 라벨을 박는 것은 함정** — 호혜는 원자가 아니라 상환기대·구속력·기한·등가성·약정의 묶음이고, "상환 구속력(soft/hard)"조차 기한·강제메커니즘·위반결과의 묶음이다(친구 구두 대여 = 기한 명시인데 강제는 사회적, 독립적으로 변함). 원자성을 끝까지 따지면 무한 후퇴 → **멈춤 기준은 "더 쪼개도 신용 신호가 안 바뀌면 멈춘다"(목적상 원자).**

신용이 *실제로* 쓰는 최소 차원은 둘뿐:
1. **청산구조** (이미 있음) — 무청산(증여=CS)을 가른다.
2. **`default판정` 불리언 — "미반환이 신용 손상 사건인가"** — 부조 vs 대여를 가른다. 부조 미반환 = 기회 미도래(신용 안 깎임, default 불가) / 대여 미반환 = 명백한 불이행(신용 깎임, default 가능).

```
증여 = 청산:무청산
부조 = 청산:미청산 + default판정:불가
대여 = 청산:미청산 + default판정:가능
fold 키 = (from, to, 자원, 청산구조, default판정)   ← 셋이 따로 쌓임
```

그 아래(사회 강제 vs 법 강제)는 신용 신호를 안 바꾸니 지금은 안 쪼갠다(떼인 친구 대여나 떼인 은행 대출이나 "안 갚는다" 신호는 같음). 그리고 **대여 default = 깨끗한 이행 신호 → 2층 건너뛰고 3층 직접**(업체 거래 MP와 같은 경로, H12 보강2). 부조=엣지(2층)·증여=CS(2층)·대여 잔액=EM(2층)+default(3층).

> 원칙: fold 키엔 **복합 라벨(호혜유형) 금지, 목적상 원자 차원만**(청산구조 + default판정). 1층에서 '부조'를 5요소로 분해했던 것과 같은 규율.

## 3. 워크드 예제 — 영희의 결혼식 (검증 결과)

입력 신호(project 출력): 부조(민수 10만·수진 5만, EM) · 함께함(CS, 비대칭) · 들러리 역할신호(H13) · 사진업체 대금(거래).

| 결과 | 값 | 의미 |
|---|---|---|
| EM net 민수↦영희(돈) | **+100,000** | 영희가 민수에게 미청산 채무 |
| EM net 수진↦영희(돈) | **+50,000** | 영희가 수진에게 미청산 채무 |
| EM 영희↦사진업체 | **0** | 거래(즉시청산) → 트러스트 흔적 없음 |
| 영희의 받은 부조 미청산 총량 | **150,000원** | 영희가 앞으로 갚을 사회적 채무 |
| CS 민수→영희 / 영희→민수 | **2 / 1** | **방향 비대칭**(B층위) 확인 |
| CS 영희→수진 / 수진→영희 | **4 / 1** | **역할신호**(들러리=+3)로 영희→수진 강함 |

검증 3건 통과:
- **① EM 청산**: 영희가 나중에 민수 결혼식에 10만 부조 → 민수↦영희 net **0으로 청산**. 수진은 안 갚아 5만 유지.
- **② 순서 무관**: 신호를 섞어 fold해도 **동일 상태**(모노이드).
- **③ 배신**: 민수→영희 유대에 −5 신호 → tie **−3(적대)**. CS가 음수로 깎임(H3 정정 확인).

## 4. 일관성 점검 (F2-6)

기존 결론과 정합:
- **H8** — 상태 = fold(신호). fold 결합법칙 성립(순서무관) → H8의 모노이드 요건 충족. 상태가 다음 이벤트를 소집하는 ↑ 루프는 미구현(다음 단계).
- **H11** — fold가 접는 건 raw 액션이 아니라 **신호**(project 출력). ✅
- **H12** — EM·CS 같은 형식 `(방향, 자원, 크기)`, 차이는 **청산 유무**(EM 상쇄 / CS 무상쇄). CS도 방향·음수 가능. ✅
- **H3** — EM = 아벨군(청산=역원), CS = 부호 있는 방향별 누적(역원 없음). ✅

스폿체크(타 사례):
- **계모임(ROSCA)** — 3인 3라운드 fold: 사이클 끝 pairwise net **전부 0**(공정 계는 완전 청산). 1라운드 후 먼저 받은 A는 net 음수(빚) = 다른 멤버가 A의 신용에 노출. 𝒲 본진과 정합.
- **중고거래(MP)** — 즉시청산 → EM·CS 항목 **0개**. 거래는 트러스트 잔액을 안 남김.

## 5. 열린 항목 (다음 단계)

- **CS 비대칭의 데이터 추정** — `tie(A→B) ≠ tie(B→A)`의 원천은 B층위(기존 유대·지위차). 대칭 사건만으론 비대칭이 안 나오므로, 디방 데이터로 그 맥락을 복원할 수 있는지가 관건(H12 검증 포인트). 못 하면 CS는 대칭으로 둔다.
- **비선형 fold** — 현재는 순수 덧셈(선형). 배신이 누적 호의를 불비례로 깎거나, 유대에 상한·감쇠(시간 망각)가 있으면 비선형이 되고 순서 의존이 생길 수 있음. 1차는 선형 유지.
- **H8 통합 루프 ↑** — 상태가 다음 이벤트를 소집하는 방향. fold(누적)는 됐고, "상태→이벤트 생성"은 미착수.
- **한 액션이 두 원장에 동시 기여** — 예: 참석은 `wedding-action-catalog`에선 EM(시간 부조, 미청산)인데 fold 예제에선 CS(함께함)로 봤다. 둘 다 맞다 — 참석은 시간을 *주는*(EM) 동시에 함께 *있는*(CS) 행위라, project가 한 액션에서 **신호 둘**(EM 시간 + CS 유대)을 뽑을 수 있다. 자원종류가 파라미터라 같은 fold가 둘을 각 원장에 누적. 어떤 액션이 단일/이중 신호인지의 정식 목록은 미정.
- **EM 자원종류와 환산 (2026-06-16)** — EM 자원은 한 가지가 아니다: **돈·물건·노동/시간·정보/기회·돌봄·향응·공간·전문성**(+ 보증/추천 — 평판을 빌려줌, 신용망 핵심 후보). 공통점 = "상대에게 비용 드는 걸 넘긴다". 돈만 단위가 명확(원)하고 나머지는 자원 간 **환산이 미정**("이사 하루 = 밥 몇 끼?"). **단 환산은 fold의 블로커가 아니다** — fold는 자원종류별로 잔액을 따로 누적하므로(자원=파라미터) 단위 있는 자원(돈·시간·횟수)은 지금도 굴러간다. 환산이 꼭 필요해지는 건 *여러 자원을 한 엣지 가중치로 집계*하거나 *3층 페이지랭크(스칼라 가중치)*로 갈 때 = **2층→3층 다리 문제**지 2층 fold 문제가 아니다.
- **CS 저장 방식 (설계 선택)** — CS 유대를 본 문서처럼 **별도 CS 원장**으로 저장할지, **EM 패턴서 추론**할지(05 H12 보강)는 미확정. 1차 모델은 추론으로 간소화 가능. 횟수형 CS(초대·참석·동석)는 수집되므로 별도 저장도 부담 적음.
- **hackathon 𝒲/Φ 연결** — EM 원장 = 𝒲(호혜-원장 카테고리)·C2(Grothendieck 군). 이 fold 상태를 그 트랙에 이어 **3층 Φ(관계망→신용 = 네트워크 전파)**로. ← 신용 모델의 입구. (신용은 잔액 크기·부조 이행률이 아니라 *그래프 전파*에서 나옴 — 05 H12(4) 정정.)
