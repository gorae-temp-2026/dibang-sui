# E2E 테스트 시나리오 — 온체인 전체 경로 검증

> **목적**: 실제 testnet에서 모든 온체인 함수가 동작하는지, SDK→컨트랙트 연결이 끊긴 곳은 없는지, 데이터가 실제로 생성·조회되는지를 빠짐없이 검증한다.
> **방법**: 3개 에이전트가 각자 지갑을 생성하고 SUI를 충전한 뒤 독립 시나리오를 병렬 실행한다. 실패 지점을 찾아 보고하고, 반복 루프로 모든 경로가 통과할 때까지 돌린다.
> **날짜**: 2026-06-22

---

## 공통 사전 작업 (각 에이전트 공통)

| 단계 | 작업 | 검증 |
|------|------|------|
| W-0 | `configureSui(TESTNET_CONFIG)` 호출 | config.packageId가 설정됨 |
| W-1 | `createJsonRpcClient('testnet')` 생성 | client 인스턴스 정상 |
| W-2 | `Ed25519Keypair()` 으로 신규 지갑 N개 생성 | 각 지갑의 address가 유효한 Sui 주소 |
| W-3 | env의 자금 지갑(FUNDER_SECRET_KEY)에서 각 지갑으로 0.1 SUI 전송 | `executeAndAssert` 성공, 각 지갑 잔액 ≥ 0.1 SUI |
| W-4 | TrustMatrix/TrustRegistry 공유 오브젝트 ID 확인 | `csMatrixId`, `emMoneyMatrixId` 가 constants에 존재하고 온체인에서 조회 가능 |

### B-7. EdgeBalance (2층 fold + 3층 전파 동시 갱신)

> edge_balance 모듈은 두 사람 사이의 쌍-국소 신뢰 잔액을 보관하고, record() 호출 시 2층 fold와 3층 TrustMatrix 전파를 한 tx에서 동시에 수행한다. SDK 빌더 없음 — 직접 PTB 구성.

| ID | 시나리오 | Move 함수 | 검증 |
|----|----------|-----------|------|
| B-7-1 | 두 사용자 사이 EdgeBalance 생성 | `edge_balance::create_edge(p0, p1)` | EdgeBalance(shared) 생성, EdgeCreated 이벤트, participants=(p0,p1) |
| B-7-2 | 생성된 EdgeBalance 조회 | `client.getObject({id: edgeId})` | p0, p1 주소 일치, balances 비어있음 |

### 검증 판정 기준

- **성공 시나리오**: `executeAndAssert`가 throw 없이 완료 + 후속 조회에서 기대 데이터 확인
- **실패 시나리오**: 트랜잭션이 Move abort 또는 실행 에러로 실패 + 에러 메시지에 기대 abort code 포함
- **조회 시나리오**: SDK 쿼리 함수가 null이 아닌 데이터 반환 + 필드값이 기대와 일치

### 반복 루프 전략

1. 각 에이전트가 시나리오를 순서대로 실행
2. 실패한 시나리오 ID와 에러 메시지를 수집
3. 실패 원인 분석 (SDK 빌더 버그 / 컨트랙트 불일치 / 오브젝트 ID 참조 오류 / 가스 부족 등)
4. 코드 수정 후 실패 시나리오만 재실행
5. 모든 시나리오 PASS까지 반복
