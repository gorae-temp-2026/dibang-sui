## 0. 전역 자원·운영 모델 (모든 Phase 공통)

### 0-1. 서브에이전트 운용
- **브리핑 필수**: 모든 서브에이전트 프롬프트 맨 앞에 `00-INTENT-CHARTER.md` 전문 + 해당 Phase 규칙을 주입한다. 브리핑 없이 스폰 금지(메모리 `spawn-agents-fully-onboarded`).
- **동시성**: 한 번에 ~10개(워크플로 cap = min(16, 코어-2)) 배치 실행. 초과분은 큐.
- **반환 규칙**: 서브에이전트는 **내용을 반환하지 않는다.** 출력 파일을 직접 `Write`하고, 통계(JSON: `{src_lines, out_lines, dropped_bytes, rule_ok}`)만 반환한다. → 메인 컨텍스트 폭발 방지.
- **에이전트 타입**: 추출(1·2·3)은 기계적 cut → `Explore`/일반 에이전트 + `effort: low~medium`. 4차(코드·문서 독해)·6차(형식화)는 판단 필요 → 표준 에이전트 + `medium~high`.
- **실행 수단**: 대량 팬아웃은 `Workflow`(pipeline/parallel)로 결정적으로 돌린다. Phase 내 파일단위는 pipeline, Phase 경계는 검증 게이트(barrier).

### 0-2. 거대 파일 처리 (청크)
- 기준: **8MB 초과 또는 2,500줄 초과** 파일은 라인레인지로 분할. 청크 크기 ≈ 2,000줄.
- 청크별 서브에이전트가 `offset/limit`로 읽어 부분 pass1 생성 → 파일별 `MERGE` 태스크가 순서대로 이어붙임(cut-only 유지, 재배열 금지).
- 해당 후보(C·E의 75M/49M/31M/23M/18M/17M/16M/15M~13M대 다수 ≈ 30~40파일).

### 0-3. cut-only 검증 방법 (1·2·3차)
- 자동: 출력의 각 라인(또는 청크)이 입력의 **연속 부분문자열**인지 substring 체크. 위반 라인 0이어야 통과.
- 분량: 단계별 잔존율 측정(2차 ≥ 1차의 50%).
- 샘플 수동: Phase당 무작위 N=5파일 육안 확인(해석/요약 흔적 없는지).

### 0-4. 진행 원장
- `_TASKS/ledger/`에 태스크별 상태(`pending/in_progress/done/failed`)·입출력경로·통계 기록. 중단·재개 가능하게.

## PHASE 1 — 1차 추출 (원문 700MB 전체 읽기 · cut-only)

> 규칙: 질의응답 + 도구/스킬 실행 + 주요 맥락만 남기고 **잘라내기만**. 해석·압축·재배열 금지. **목표는 700MB 전부 읽기.**

- **입력**: 160개 원문 `.jsonl` · **출력**: `01-extract-pass1/<code>/<sessionid>.md`
- **추출 대상(남길 것)**: ① 사람 user 메시지 전문(verbatim), ② assistant 텍스트 응답 전문, ③ 도구/스킬 호출 라인(이름+핵심 인자)과 결과의 맥락 머리/꼬리. **버릴 것**: base64·이미지·거대 파일덤프·전체 diff 본문(머리/꼬리+`…[N cut]…`만).
- **태스크 구성** (P0.3이 실제 생성):
  - `P1.EX.001 … P1.EX.160` — 파일당 1개(소·중형). 각: 헌장+Phase1 규칙 브리핑 → 파일 읽고 pass1 작성 → 통계 반환.
  - `P1.EX.NNN.c01…cK` — 거대파일 청크(≈30~40파일 × 평균 6~10청크).
  - `P1.MERGE.NNN` — 청크 이어붙이기(순서 보존, cut-only).
- **자원**: pipeline 동시 ~10. 거대파일은 청크 팬아웃 후 머지.
- **검증 게이트 `P1.V`**:
  - 커버리지: 160/160 출력 존재.
  - cut-only: 자동 substring 체크 위반 0 + 샘플 5파일 육안.
  - 목표확인: 원문 총 bytes 읽힘 로그(읽기 커버리지 100%).
- 추정 태스크 수: ≈ 160 + 청크 ~250 + 머지 ~35 + 게이트 = **약 230~280**

## PHASE 2 — 2차 추출 (진짜 불필요만 제거 · 50%+ 잔존 · cut-only)

> 규칙: pass1에서 **진짜로 필요 없는 맥락만** 제거. 여전히 잘라내기만. **pass1 분량의 50% 이상 남아야 함.**

- **입력**: `01-extract-pass1/` · **출력**: `02-extract-pass2/`(동일 구조)
- **제거 후보**: 순수 노이즈(반복 빌드로그, 동일 에러 재시도, 환경 덤프, 의미 없는 확인응답). **제거 금지**: 사용자 발화·결정·지적·피드백·의도가 담긴 모든 것.
- **태스크**: `P2.EX.001…160`(pass1 파일당 1개). 헌장+Phase2 규칙 브리핑.
- **검증 게이트 `P2.V`**: 파일별·전체 잔존율 ≥50% 확인 + cut-only(pass2 ⊂ pass1) substring 검증 + 샘플 육안(사용자 발화 누락 0).
- 추정: ≈ 160 + 게이트 = **약 165**

## PHASE 3 — 3차 추출 (모순·레거시 제거 · 일관 맥락만 · cut-only)

> 규칙: 모순되거나 폐기(레거시)된 맥락 제거, **연결성 있어 일관된 맥락만** 남김. 잘라내기만. 가장 정제·잊으면 안 되는 핵심.

- **입력**: `02-extract-pass2/` · **출력**: `03-extract-pass3/<cluster>.md`
- 이 단계는 **교차 파일 추론** 필요 → 주제/시간축 클러스터링 후 처리.
- **태스크**:
  - `P3.CLUSTER` — pass2 전체를 주제·기능·결정 단위 스레드로 군집화 → 클러스터 인덱스(어느 파일·구간이 어느 스레드인지 매핑).
  - `P3.EX.{cluster}` — 클러스터별: 나중 발화가 앞 발화를 뒤집은 경우 **최종 일관본 구간만 남기고** 폐기본 cut. 모순쌍·폐기근거를 `P3.contradictions.md`에 기록(어느 구간을 왜 남겼나). 재작성 금지.
- **자원**: 클러스터 수 ≈ 20~40, 클러스터당 1 서브에이전트(medium).
- **검증 게이트 `P3.V`**: 모순목록 해소 확인, 연결성(스레드 단절 없음) 확인, cut-only 검증.
- 추정: ≈ 1(군집) + 20~40 + 게이트 = **약 25~45**

## PHASE 4 — 4차 가공 (현 프로젝트 문서·코드 중요부 덧붙이기 · 더 방대해짐)

> 규칙: 3차를 **건드리지 않고**, 관련 위치에 현 dibang-sui 문서·코드의 중요부를 **덧붙임**. 압축요약 아님. 분량 증가.

- **입력**: `03-extract-pass3/` + 현 프로젝트 전체 · **출력**: `04-process-pass4/<cluster>.md`(pass3보다 큼)
- **문서 소스**: `_onboarding/*`(12개), `_architecture/*`, `_scenario/*`, `_code_convention/*`, `_research/*`, `CLAUDE.md`(전역+프로젝트).
- **코드 소스**: `contracts/dibang_wedding`(Move), `apps/api`(Go), `apps/dibang-wedding`, `apps/guest-web`, `packages/*`.
- **태스크**:
  - `P4.DOC.{area}` — 문서영역별 서브에이전트가 중요부 추출(출처경로 명시).
  - `P4.CODE.{subsystem}` — 코드 서브시스템별 중요부 추출(파일:라인 인용).
  - `P4.MERGE.{cluster}` — 추출 중요부를 pass3 클러스터의 맞는 위치에 덧붙임(3차 원문 보존).
- **자원**: 문서영역 ~6 + 코드 서브시스템 ~5 + 클러스터 머지 ~20~40. medium~high.
- **검증 게이트 `P4.V`**: 분량이 pass3보다 증가, 덧붙인 내용 전부 출처경로 보유, pass3 구간 그대로 존재(뼈대 보존) 확인.
- 추정: ≈ 11 + 20~40 + 게이트 = **약 35~55**

## PHASE 6 — 6차 가공 (한 폴더·여러 규격 형식화 + 요약 · 뼈대 보존)

> 규칙: 여기서만 실질 압축·연결 허용. 단 5차의 주요 뼈대가 출력에 모두 추적되어야 함. 결과도 방대.

- **입력**: `05-process-pass5/` · **출력**: `06-process-pass6/`(여러 파일)
- **산출 규격(예시, 확정은 P6 착수 시)**:
  - `INDEX.md`, `SUMMARY.md`(요약본)
  - `HANDBOOK.md`(서사형 온보딩 핸드북)
  - `DECISION-LOG.md`(결정+의도 연대기)
  - `INTENT-TIMELINE.md`(의도 변화 타임라인)
  - `FAQ.md` / `GLOSSARY.md`
  - 필요 시 프로젝트별(A/B vs v3계열) 분권
- **태스크**: `P6.FMT.{규격}` + `P6.SUMMARY` + `P6.TRACE`(5차 뼈대→6차 산출 추적 매트릭스) + 게이트.
- **자원**: 규격당 1 서브에이전트(high), 마지막 통합·추적은 메인.
- **검증 게이트 `P6.V`**: 추적 매트릭스로 5차 주요 스레드 100% 매핑(뼈대 소실 0) 확인 + 요약본 존재.
- 추정: ≈ 6~8 + 게이트 = **약 10**

## 의도 이탈 방지 체크 (모든 태스크 공통 선언)
1. 나는 `00-INTENT-CHARTER.md`를 읽었다.
2. 내 단계의 "절대 규칙"(cut-only / 덧붙이기 / 표현만 / 뼈대보존)을 안다.
3. 나는 내용을 반환하지 않고 파일로 쓰고 통계만 반환한다.
4. 의심되면 임의 판단 말고 멈추고 보고한다.
