## 큰 그림 — 4층 사다리 (분류 → 누적 → 환산 → 상품)

```
1층  분류    모임 → 타입/프로필            classify           [집합론]   ✅ 완성(리서치)
2층  동역학  액션을 쌓아 관계·집단 형성     상태 = fold(신호)   [집합론+대수] ✅ 골격(fold·시뮬)
3층  신용    관계망 → 신용 점수            Φ: 관계 → 신용      [범주론]   🔧 시제품(PageRank+행동신호)
4층  금융    신용 → 실제 상품(대출·보증)    (서비스)                      ⬜ 미구축 ← 해커톤에서 연결
```
- 핵심 순서: **분류 → 누적 → 환산 → 상품.**
- 현재 위치: 1·2층은 리서치 완성/골격, 3층은 시제품(`sim.mjs`), 4층(DeFi 상품)은 미구축 — **이번 해커톤에서
  3·4층을 실제로 잇는 것이 목표**(정식 Φ·요네다·정량모델은 Sui 𝒲 트랙 소관).

## 왜 결혼식인가 — "네트워크 스냅샷 생성기"

결혼식은 **5개 축 값의 유일 조합**(통과의례 × 전체소집 × 부조 × 공식기록 × 일생1회)으로, 한 사람의
신뢰네트워크를 **일생에 한 번 전수 샘플링**하는 장치다. 즉 결혼식 하객·축의·방명록 데이터는 그 사람의
관계망을 통째로 찍는 스냅샷이라, 신뢰 잔액의 가장 농밀한 진입점이다. (장례식·돌잔치 등으로 확장 가능.)

---

## 가장 중요한 설계 원칙 — raw 액션 저장 vs 신뢰 투영 계산 (분리)

`wedding-action-catalog.md`(2026-06-15 정정)의 핵심:

- **raw 동사(action_type)** = 사람이 한 *기계적 행위*(해석 없음). **이것을 저장(온체인)한다.**
  - 예: `돈 건넴`, `옴(참석)`, `말함`, `서명함`, `글·영상 보냄`, `초대함`, `노래함`, `서약함`, `선언함`, `동석`, `안아줌` …
- **해석(부조·거래·축하…)** = project가 **`(raw 동사 × event_type × 역할쌍)`** 으로 *계산*하는 출력(signal).
  **저장하지 않는다**(규칙으로 계산).
  - 예: 같은 "돈 건넴"이 (하객→혼주)면 **부조(EM○)**, (혼주→업체)면 **거래(EM✗·신용○ 이행률)** 로 갈린다.

→ **온체인엔 raw 액션 + 맥락(누가·누구에게·어느 이벤트·자원·금액)을 쌓고, EM/CS/신용 신호는 해석 규칙으로
계산한다.** 이래야 같은 행위의 다중 해석/규칙 진화가 가능하다.

## 신호의 3층위 (단일 ○/△/✗는 과부하였음 → 분할)

행위가 만드는 신뢰 신호를 **Fiske 4모델** 기반 3층위로 나눈다:

- **EM (Equality Matching) — 2층 호혜 잔액**: 아벨군(net=청산 가능). 부조·참석·축전·서명·화환·공동부조·감사.
  미청산 잔액을 지원하는 유일한 모델 → **신용 함자 Φ의 정의역**(Pacioli 복식부기 = Grothendieck 군과 접속).
- **CS (Communal Sharing) — 2층 유대 강도**: 방향 있는 유대, 무청산·비대칭. 초대(사전)·참석·동석·안아줌·
  축하인사·서약(부부)·사진·축가. 돈 0이어도 CS○ — 친밀의 직접 신호.
- **신용 (3층 재료)**: 부조 이행 · **업체 거래 이행률(MP)** · 약속/계약 이행. (즉시청산 거래는 EM 잔액엔 ✗지만
  "제때 냈나"는 신뢰성의 직접 증거 → 3층 ○.)
- (AR=서열/권위: 좌석·축사 순서·성혼 선언 등 — 현재는 구조 신호로 보류.)

> 신호원은 둘: **행위 신호**(부조·방명록 — 주로 EM) + **역할·관계 신호**(누구를 초대·주례·사회자로 세웠나 —
> 주로 CS). 후자가 EM으로 안 잡히는 친밀을 보완.

## 2층 fold — 신호를 상태로 누적

- `상태 = fold(신호들)`. **fold는 부호 합산(순서 무관).** EM 원장(net 아벨군·청산) + CS 원장(방향별 유대·
  무청산·비대칭).
- **fold 키 = (from, to, 자원, 청산구조, default판정)** — 증여(무청산)·부조(미청산+default불가, 미반환=갚을
  *기회* 미도래라 신용 안 깎임)·대여(미청산+default가능, 미반환=불이행)를 분리. 멈춤 기준: "더 쪼개도 신용
  신호가 안 바뀌면 멈춘다."
- 결과 = **가중 관계 그래프**(엣지값 = 두 사람 사이 누적 신뢰).

## 3층 신용 전파 — 그래프 → 노드 신용 점수 (`09-credit-propagation.md`, `sim.mjs`)

4단계 파이프라인:
1. **자원별 동질 그래프 분리**: 부조 그래프 · CS 그래프 (대여는 sparse → 행동신호로).
2. **자원별 PageRank**(무방향, d=0.85) → 부조점수 · CS점수 (0~1).
3. **행동 신호 직접 투입** → 이행점수(대여 갚음=1.0 / default=0.2 / 무기록=0.7). (전파 안 함 — 본인 노드 직접.)
4. **점수 합산(용도별 비중)** → **최종신용 = 0.5·부조 + 0.3·CS + 0.2·이행.**

- ①② = 네트워크 위치("누구와 엮였나" 전파), ③ = 본인 행동("의무 지켰나").
- **H12: 신용 = 본인 EM 행적 + 본인 MP 이행률 + 네트워크 전파(PageRank).** "신용 = 네트워크 위치"(요네다 직관).
- **유대(CS) ≠ 신용**: 유대는 *관계*의 속성, 신용은 *사람*의 속성. 신용은 유대의 *양*이 아니라 *누구와
  엮였나*로 정해진다(건달과 깊은 유대는 신용에 −, 대부호와의 유대는 +) → 고유벡터 중심성=PageRank의 신뢰 버전.
- 시뮬(10명×10년) 검증: 망 중심(부조 모인 사람)↑, CS 허브↑, 이행 좋음↑, default↓·고립↓ — 직관과 정합.
- **주의: 결정값(무방향·default 0.2·비중 0.5/0.3/0.2)은 first-cut 임의.** 정식 Φ(=S∘L)·요네다·MEC 정량모델은
  Sui 𝒲(하커톤) 트랙에서 정밀화. 실데이터 보정 필요.

## 4층 금융(DeFi) — 신용 → 상품 (미구축, 해커톤 목표)

한 지갑에 연결된 모든 관계의 신뢰 잔액 → 그 지갑의 신용 점수 → **대출 한도·결제 보증·이자율** 등 DeFi 상품의
재료. 무담보/씬파일(thin-file) 신용을 **사회적 관계 신뢰**로 푸는 것이 "beyond traditional DeFi"의 핵심(09 정렬).

## 온체인 데이터 요구사항 (이 모델이 요구하는 스키마)

데이터 모델 v0.1 (H11) — **7개 엔티티**. action = fold 원자, 나머지는 그 해석에 필요한 맥락.

```
entity(id, kind=사람|집단)                                ← 그래프 노드
event_type(type_id, 이름, 11축 프로필)                    ← 결혼식 등 타입 카탈로그
event(id, type_id, 시간창, 장소)                          ← 인스턴스(규모는 participation 수로 파생)
participation(event_id, entity_id, role_id)               ← 참가 = (사람, 역할)
role_type(role_id, 이름=혼주/하객/주례/사회자/공급자…)
action(id, event_id, actor, action_type, target?, payload, timestamp)  ← fold 원자
action_type(action_type_id, 이름=돈건넴/옴/서명/초대…)     ← id·이름뿐. 자원·극성 칼럼 없음
```

핵심 설계 결정(연구 확정):
1. **해석값(자원·청산·극성·공개)은 저장하지 않는다.** `project(action_type × event_type × role)`로 계산.
   부정 행위(배신·잠수)도 그냥 독립 action_type — **±극성조차 스키마에 없고 규칙이 매긴다.**
2. **모든 action은 event_id 필수**(1:1 DM·송금도 "퇴화한 2인 event"). actor·target은 사람|집단 다형,
   **target은 nullable**(관람·입장은 대상 없음).
3. **역할(participation.role_id)이 fold의 *방향*을 만든다.** 역할이 없으면 부조가 무방향이 된다 → 누가
   혼주이고 누가 하객인지가 있어야 `하객→혼주` 신뢰 엣지가 나온다. action의 actor/target만으론 부족.
4. **모든 활동·관계 기록은 그 지갑에 soulbound(SBT, key-only)** — 신용 무결성(VISION §4).
5. **민감정보(이름 등) 온체인 평문 금지**(VISION §7). 신뢰 신호는 비민감 상호작용으로.

> **온체인/오프체인 경계**: 온체인 = raw 액션 + participation(역할) + event/entity (= SBT가 든 *사실*, 불변·검증).
> 오프체인(또는 별도 모듈) = `project`(해석) · `fold`(EM/CS 누적) · `PageRank Φ`(신용). **후자의 결정값
> (비중·default 페널티)은 임의·튜닝 대상이라 온체인에 박으면 안 된다.** 이미 만든 컨트랙트(guestbook/cash_gift/
> ium 등)는 이 "액션 원장" 관점으로 재정렬 + SBT(key-only)로 맞춘다(`06-SUI-ONCHAIN-DIRECTION.md`, D15).
