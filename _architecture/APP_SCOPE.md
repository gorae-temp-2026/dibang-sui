# 앱별 기능 경계

## 경계 기준

**전환 퍼널이냐 서비스 본체냐**가 두 앱을 나누는 기준이다.

- 비로그인 전환 퍼널 (공유 링크/QR 착지 → 참여 경험 → 웨딩 라운지 진입 유도) → **Guest Web**
- 로그인 서비스 본체 (식별된 Host/Guest의 모든 Use Case) → **Dibang Wedding**

로그인 여부는 이 경계를 보조 설명한다(퍼널은 대체로 비로그인, 본체는 로그인). 다만 **로그인 여부가 유일·절대 기준은 아니다.** Guest Web은 비로그인 상태에서 방명록·메시지뿐 아니라 **축의(현금 송금) 플로우까지** 책임진다. 즉 "비로그인이면 무조건 최소 행위만"이 아니라, "전환 퍼널 단계 전부가 Guest Web, 그 이후 식별된 사용자의 본체 기능이 Dibang Wedding"이다.

> 이전 정의("로그인 여부가 두 앱을 나누는 유일한 기준", "Guest Web은 청첩장+방명록만")는 초기 기획 기준이었다. 축의 경험을 Dibang Wedding(로그인)에서 Guest Web(비로그인)으로 옮기기로 방향을 바꾸면서(2026-05-18 확정) 이 문서를 현실에 맞게 재정의했다.

## Guest Web (guest-web)

**역할**: 비로그인 전환 퍼널. 공유 링크/QR의 착지점. 두 갈래 퍼널을 거쳐 **웨딩 라운지(Dibang Wedding) 진입을 유도**한다.

### 퍼널 A — 모바일 청첩장 퍼널

**가능한 행위** (로그인 없이):
- 모바일 청첩장 보기
- 청첩장 하트
- 라운지 티저 확인

**종착**: 청첩장 조회 응답(InvitationPublic)에 포함된 라운지 이름·방문자 수로 라운지의 존재감을 노출하여 웨딩 라운지 진입을 유도한다.

### 퍼널 B — 방명록·메시지·축의 퍼널

**가능한 행위** (로그인 없이, 순서):
- 누구측(6슬롯) 선택
- 관계/이름 입력 → 참석 기록 생성(GuestbookEntry)
- 금액 선택
- 송금(축의) → CashGift 생성
- 축하 메시지 작성(GuestbookMessage) 또는 하트
- 완료

**종착**: 완료 화면에서 **'사진 공유'를 후킹**으로 사용하여 웨딩 라운지 진입을 유도한다. (참여한 하객이 사진을 공유하려면 라운지로 들어가야 하므로 자연스러운 진입 동기가 된다.)

### 도메인 모델 매핑

- `ViewMobileInvitation` — 청첩장 조회 + visited_count 증가
- Heart — heart_count 증가 (별도 Command 없음, 카운터 증가)
- `CreateGuestbookEntry` — 비로그인 방명록/참석 기록 작성 (guest_id optional)
- `CreateCashGift` — 비로그인 축의 기록 작성
- `CreateGuestbookMessage` — 비로그인 축하 메시지/하트 작성

### 웨딩 라운지 진입 유도

두 퍼널의 종착은 모두 웨딩 라운지다. 라운지 진입 자체(LoungeCheckIn 생성, 라운지 열람)는 식별된 사용자의 행위이므로 **Dibang Wedding의 라운지 진입 경로(`/lounge/{loungeId}/enter`)로 이동**시킨다. 해당 경로는 로그인(AuthGuard) 뒤에 있으므로 비로그인 하객은 자연히 로그인 퍼널을 거쳐 라운지로 들어간다.

### 축의 소유

- **비로그인 축의 = Guest Web** (퍼널 B). 현장 하객이 QR로 진입해 식별 없이 축의·메시지를 남기는 경험.
- **로그인 상태 메시지·축의 = Dibang Wedding** — QR 페이지를 통한 로그인 사용자용. **추후 구현 예정**(현재 미구현).

## Dibang Wedding (dibang-wedding)

**역할**: 서비스 본체. 로그인한 Host와 Guest 모두 사용. 전환 퍼널 이후의 모든 식별된 행위.

### Host 기능
- 결혼식 만들기 (Wedding + Lounge + GatherPlace + Invitation 복합 생성)
- 결혼식 정보 수정
- 모바일 청첩장 수정, 공유 링크 관리
- 공지사항 발행/수정
- 인테리어 아이템 추가, 배치/해제

### Guest 기능 (로그인 상태)
- 웨딩 라운지 입장 (LoungeCheckIn 생성)
- 모이가모인곳 (MoiGatherPlace) 조회, 방문 모이 열람
- 방명록 조회, 공지 조회
- (추후 구현 예정) QR 페이지를 통한 로그인 상태 메시지·축의

### 공통 기능 (Host + Guest)
- 모이 꾸미기 (아이템 장착/해제)
- 모이 아이템 선물
- 이음 생성/조회/삭제
- 프로필 조회/수정

### 도메인 모델 매핑
`RegisterUser` + `CreateMoi` 이후의 **모든 Use Case**가 Dibang Wedding에 해당한다.

## 범위 밖 (참고)

- **디스플레이**: 하객이 QR에서 보낸 메시지를 현장 디스플레이에 표출하는 기능. 추후 별도 구현 예정이며 본 경계 문서의 두 앱 범위에는 포함하지 않는다.

## 요약

```
                  ┌─ 퍼널 A: 모바일 청첩장 + 하트 + 라운지 티저 ─┐
Guest Web ────────┤                                              ├──→ 웨딩 라운지 진입 유도
(비로그인 퍼널)    └─ 퍼널 B: 방명록·메시지·축의 → '사진 공유' 후킹 ─┘            │
                                                                              ▼
                                                       Dibang Wedding /lounge/{loungeId}/enter
                                                       (로그인 후 라운지, 모이, 이음, 관리 등 본체 전부)
```
