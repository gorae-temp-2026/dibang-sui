## [2026-05-05 03:10] REST 엔드포인트 분류 기준 불일치

**상황**: API_ENDPOINT_MAP.md, api-contract.yaml(주석 블록), api-contract.yaml(tags)를 작성하면서 `/users/me/weddings`를 매핑 문서에서는 Weddings에, spec에서는 Users 블록에 배치. `/users/me/iums`도 매핑 문서에서 Iums에 배치. 같은 `/users/me/*` 패턴에 대해 세 산출물의 분류가 모두 달랐다.

**지적**: "왜 이런 실수를 했지?" + "같은 패턴인데 또 다르게 배치한 이유는?"

**원인**:
1. 분류 기준을 먼저 정의하지 않고 작성 시작
2. 매핑 문서는 "기능 중심"으로, spec은 "path 구조 중심"으로 각각 다른 기준을 무의식적으로 적용
3. Task #4~#8을 한 번에 처리하면서 Task #3 산출물과 교차 검증 생략
4. 리뷰어 프롬프트에 "산출물 간 분류 일관성" 점검을 넣지 않음

**교훈**:
- REST 엔드포인트 분류의 유일한 기준은 **path의 첫 번째 세그먼트(루트 리소스)**
- `/users/me/*` 패턴은 전부 Users 섹션. tags로 관련 리소스를 보조 표시할 뿐
- **매핑 문서 섹션 = spec 주석 블록 = 첫 번째 tag** — 이 세 곳이 반드시 일치해야 함
- 여러 산출물을 작성할 때는 분류 기준을 먼저 통일하고, 완료 후 교차 검증
- 리뷰어에게 "산출물 간 일관성"을 명시적으로 점검하라고 지시할 것

## [2026-05-05 04:30] 설계 문서에서 축약형 사용 금지

**상황**: API_ENDPOINT_MAP.md에서 소비자 칼럼을 "DW", "GW"로, API_CONSUMER_NEEDS.md에서 읽기/쓰기를 "R", "W"로 축약해서 작성했다. 사용자가 이 축약형을 사용한 적도 없고 동의를 받은 적도 없다.

**지적**: "축약하지 말고 그대로 적어줘" + "Read, Write 등 영어로 적어줘"

**원인**: 테이블 칼럼 폭을 줄이려는 습관으로 임의 축약. 문서의 가독성을 에이전트 기준으로 판단하고 사용자 기준을 무시했다.

**교훈**:
- 기획 문서, 설계 문서를 작성할 때 **축약형을 절대 사용하지 않는다**
- 사용자가 먼저 축약형을 사용했거나, 명시적으로 동의한 경우에만 축약 가능
- 테이블이 넓어지더라도 원문 그대로 적는다 (Dibang Wedding, Guest Web, Read, Write 등)
- 이 규칙을 어기면 엄벌에 처한다
